The Gamepad Specification

Below, you will see the specification that has been previously described. The first page is for people using the Android APIs, the second is for Unity developers.

As of this writing, this specification takes into account over 25 different controllers from the major consoles to all major third party manufacturers. Moreover, with Google and NVIDIA recommending this specification to developers, as well as controller manufacturers, that number will only continue to grow.

Take special care to read all of the notes here, as they are important to fully implement the specification.

The Specification using Android APIs

The representative gamepad below shows an amalgam of standards brought together to simplify input handling for the developer. Please note certain controllers can issue multiple events for a single action which can safely be ignored, except the triggers — see the notes in that section. The developer only needs to handle the events listed here.

Notes
  • Handling all cases — Notice the DPAD and Triggers require you to handle more than one value. For Triggers in particular, you can’t assume that you will only ever give one value for one input like other controls. So a controller that outputs AXIS_RTRIGGER may also output AXIS_GAS with the exact same values. See the pseudo-code in the bottom of The Shoulder Buttons section.
  • Handling NO analog/MotionEvents — If, and only if, you are NOT going to handle any MotionEvents – in other words, you will only handle the DPAD buttons — you can ignore handling AXIS_HAT values for the DPAD. Android will generate the proper DPAD equivalents for AXIS_HAT and AXIS_X/Y for you.

The Specification in Unity

For the broadest range of plug and play support, we recommend using this controller as your default. If possible, it is also recommended that you use a controller setup screen. This is an Options screen, in which users assign controller functions to application functions. There are several Unity plug-ins available to help you along.

Unity Notes
  • General recommendations — We recommend using this controller as your default. This should give the best "it just worked" experience for the user. However, since Unity does controllers in a specific way, we still recommend adding a controller setup screen to back stop other controllers. Consider adding a dialog after detecting a controller that tells the user, "If you’d like to customize your controller, please see the setup screen in Options."
  • Shoulder Buttons — Please read the documentation in the “Unity API Specifics” section for handling the spec defined in the “The Shoulder Buttons” box. Keep in mind the SHIELD controller name, NVIDIA Corporation NVIDIA Controller v01.01, is an NVIDIA standard and something upon which you can rely. “v01” denotes a family of controllers that have above controller style and its outputs as its base. We recommend that any checks are made against just the major version, NVIDIA Corporation NVIDIA Controller v01.
