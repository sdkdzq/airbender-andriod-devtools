Reports

When you have completed sampling your application, Tegra System Profiler will spend a few seconds preparing the results and will then display a sampling report, which is automatically saved on disk into a .qdrep file. The main purpose of this report page is to show per-function CPU usage.

The report page consists of three views: the Analysis Summary, the Function Table View, and the Symbol Resolution Logs. The Function Table View contains an interactive timeline view on top, and a function table below.

The function table can work in three modes: the Top-Down View, the Bottom-Up View, and the Flat View.

To switch between these views, use the drop-down menus at the top of the window.

Pressing the Filter button at the top brings up a pop-up menu, which can help you narrow your results to only the threads you want to view.

Timeline

The upper pane of the report page is the timeline, which provides information over the timespan of the capture.

CPU graphs have sub-pixel information, so each graph has 2 colors.

  • Black = average
  • Gray = maximum

The top row shows processor utilization, the next several child rows show core utilization. Each core has a color code which correlates with the threads below.

In the thread group, each thread has 2 rows.

  • The first row is a graph, which shows how well the thread is utilizing the time it is provided on the CPU.
  • The second row is color coded, showing you the “core occupancy.” This shows which core it is occupying during that point in time. Due to sub-pixel information, there may be multiple color at a particular time coordinates.

To zoom in on the timeline, left-click, drag, and release.

Here you can choose whether to zoom, filter, or zoom and filter.

If you already have a selection, you can undo a zoom, remove the filter, or both.

Function Table View

The lower pane of the report page shows the various function table views.

Top-Down View

The top-down view shows the call path to functions in your application sorted by the total amount of CPU each call path took. The Self column shows the CPU usage for calls to the function itself, while the Total column shows the CPU usage for calls to this function and all functions it called.

The last column shows a graph of when each function was active during the sample. Black bars indicate time used by the function itself (the Self column) while gray bars indicate time used by calls made to other functions.

The colored bar graphs behind each Self and Total cell provide a quick visualization for that value. The color of bar graphs is derived directly from the value, while the width of the bar graph is proportional to the largest value in the column.

Note that the top-down view does not group multiple calls to the same function if they occur through different call paths. Further, functions that do little work themselves (such as VM_Call or RE_EndFrame) appear high in the list if they call other functions that do a lot of work. This yields an incomplete picture of which functions need to be called less (or rewritten to do less work) in order to improve performance. To find out which functions contributed most to CPU usage, regardless of how they were called, use the Bottom-Up View.

Bottom-Up View

While the top-down view shows a hierarchy of call paths to functions, the bottom-up view shows how much each function contributed to overall CPU usage, broken down by call path.

Note that in the bottom-up view, the Self and Total numbers displayed for rows that are not at the "root" of the tree correspond to a breakdown of the values at the root of the tree. They explicitly do not refer to the usage of that function.

Also note that items at the root of the bottom-up view with no children (e.g., memcpy) represent function calls that were sampled but had no call stack (or at least not one that could be determined by Tegra System Profiler).

Flat View

The flat view shows how much time a function was present anywhere on the call stack during the profiling session. This way, it becomes possible to easily observe how CPU utilization is distributed among different high-level parts of the application (e.g., graphics or physics simulation).

The Stack column specifies how much time that particular function was present on the call stack. Other columns are similar to those in the bottom-up view.

Analysis Summary

The Analysis Summary report gives a simple overview of your Tegra System Profiler session, including total number of threads, number of samples collected, most active functions, module activity, thread activity, and other high-level information.

Symbol Resolution Logs

The Symbol Resolution Logs view contains a detailed log of which libraries were picked to resolve symbols.
